Chapter 3 HTML

Strongly structured content for a weakly structured web.

History

Read more history in “A Brief History of Markup”, from the excellent HTML5 for Web Designers.

HTML is a language with a fraught history; competing specifications, implementations that compete with, ignore, or add to the spec, and fighting within the standards group itself. It is not a simple tale. In the interest of brevity, I will share here only the good portions:

  • Sir Tim Berners-Lee, inventor of the World Wide Web, created a document called “HTML Tags” in 1991. This never became a formal specification.
  • Not long after, the IETF created the first HTML spec: HTML 2.0.
  • The IETF soon handed the spec over to the W3C, which kept updating HTML all the way to HTML 4.0.
  • The W3C, wanting greater consistency between HTML and XML (another language they managed), created XHTML 1.0, a stricter variant of HTML 4.0.
  • The W3C then published XHTML 1.1, and all hell broke loose. Children cried in their beds, the streets ran red with the blood of the innocent, and some backwards compatibility was lost.
  • The W3C then began work on XHTML 2.0, unsatisfied with the carnage of previous versions.
  • Contributors to the W3C, positively miffed that the violence had done so little to simplify web app development, created the WHATWG and began working on HTML5.
  • The W3C continued with XHTML 2.0, for some time, eventually starting their own ‘HTML 5’ specification.
  • The WHATWG and W3C worked on ‘HTML5’ and ‘HTML 5’ for some time, eventually agreeing to work together on one HTML5 spec.

As you can see, it’s a lengthy story, but the thing to take from it all is this: HTML is a distributed collaboration between a lot of different people with their own goals. It develops slowly but surely, and changes are constantly happening.

Sounds fun right? Let’s learn how it works.

Exercise 3.1

What is the difference between HTML, XML, and XHTML?

Answer

  • HTML is the language websites are built on, and is used specifically for hypertext documents (simply put, documents that have links to other documents and can be navigated by an end-user). It is purposefully lax with its rules in the interest of properly rendering as many websites as possible.
  • XML is a language for structured data, designed specifically to be extended. For example, .docx files (Word documents) are simply a variant of XML made by Microsoft. It is very strict with its rules in the interest of ensuring all data is properly structured.
  • XHTML is a language that began as an effort to fix the issues with HTML, which included making it stricter and making it closer to XML in style.

Tags

What is a Tag?

HTML is a language for structuring content on the web. As such, its main design concern is how to properly indicate structure in a clear and consistent manner. It accomplishes this with tags (also called elements). Here are some example tags:

<em></em>

Most HTML tags come in pairs, an opening tag and a closing tag. Everything between those two tags is said to be ‘contained’ by those tags. To understand this, imagine that each pair of tags is a box. Anything between the tags is within the box. In the end, your HTML document should be a collection of boxes within boxes, each box labelled properly to indicate what is inside of it. In other words, every HTML element should provide both structure and meaning to the document it’s in.

For example, the <em> tag indicates emphasized text, like so:

<em>I should be emphasized</em>

In the code above, the words between the <em> tags should be emphasized (like the word “emphasized” just was).

There are some HTML tags that do not have a closing tag. These tags, called “void elements”, can’t contain other tags. Think of them as empty boxes.

There are also some tags that exist only to provide structure without meaning, or meaning without structure. For example the <div> tag indicates a division in the page, but without an ID or class name (which we’ll get to later), it indicates nothing about the things it contains. Similarly, the <link> tag has semantic importance (it indicates that a stylesheet is being imported), but it doesn’t directly affect the structure of the page.

Get a full list of HTML tags from the MDN.

There are many tags in HTML, each with their own specific use-case, but a basic website can be built with only a small subset of them. In this chapter we will cover those basic tags, and leave the discovery and use of more specialized tags up to you.

Basic Document Tags

So you know what tags are, but how are they used? Take a look at this basic HTML page I’ve made:

<!DOCTYPE html>
<html>
    <head>
        <meta charset="utf-8">
        <title>This Is A Title</title>
    </head>
    <body>
        <p>This is a paragraph.</p>
    </body>
</html>

There are some tags that don’t need a closing tag. Find out more in this helpful StackOverflow conversation.

The first thing you may notice is that every tag except the <!DOCTYPE html> tag has an accompanying closing tag. This is true for almost every tag in the HTML language. When it’s not we’ll point it out.

The first tag is the <!DOCTYPE html> tag. This tag indicates to the browser that the document is using the HTML5 specification. There are quite a few different possible doctypes, but modern documents often use the HTML5 one. The <html> tags enclose all the page contents. All HTML pages must have these tags.

Read more about HTML character encodings on the W3C page about it.

Next there’s the <head> tag. The head of an HTML document contains all the stuff that’s not displayed for the user. This includes things like the character encoding (which tells the browser how the document is encoded, and is indicated using <meta charset="utf-8"> for UTF-8 encoding) and the page title (shown above with the <title> tags), as well as CSS stylesheets, metadata, and pre-loading Javascript.

Then you have the <body> tags. The body of an HTML document contains the data that should be displayed for the user. This is where the page’s content goes. Inside the body of this document is a little paragraph, indicated by the <p> tags.

As you can see, even a simple page has a number of components. At its core HTML is about giving structure to data. Some of that data is for machines: like where the page’s stylesheets are. Other data is for the user, like the name of the site, or the article you’re currently trying to read.

But how do you make an HTML document? It’s easy: just create a file with the .html extension, and open it in your favorite web browser.

Exercise 3.2

Create a file called index.html and open it up in your browser. Copy the code from above into the file and see what happens to the page. What happens if you add another set of <p> tags below the current ones?

Answer

You should see another paragraph appear below the current one in the browser when you refresh the page. That’s how easy it is to add content to an HTML document!

Before we move on, let’s review:

  1. All HTML documents need to have a valid doctype. If they don’t then browsers have a tendency to freak the geek out.
  2. The <html> tags must contain everything but the doctype in all HTML documents. Seriously, this is easy, just don’t forget.
  3. The <head> of an HTML document is the place for stuff that’s not displayed for the user.
  4. The <body> of an HTML is the opposite. It’s where the displayed content of the page goes.

Content Tags

So those are the basic tags needed to create an HTML document. Next let’s look at content tags: the tags used to markup your page’s content. We already covered the <p> tag, so let’s start with lists. There are two major types of lists in HTML, ordered — represented by the <ol> tag — and unordered — represented by the <ul> tag. Each type of list should contain any number of list item elements, which use the <li> tag. Here’s an example:

<ol>
    <li>This is the first ordered list item.</li>
    <li>This is the second.</li>
    <li>This is the third!</li>
</ol>

<ul>
    <li>This is the first unordered list item.</li>
    <li>This is the second.</li>
    <li>This is the third!</li>
</ul>

Lists are used for exactly what you’d expect: listing things! Try making some lists in the HTML document you made in exercise 3.2 and see what happens.

Another important set of content tags are the <blockquote> and <q> tags. <blockquote> is for large quotes that should be displayed in their own block of content. <q> is for quotes that should be displayed inline with other content. Here is an example of each of them:

For an explanation of the blockquote code read this article from A List Apart.

<figure class="quote">
      <blockquote>A penny saved is a penny earned.</blockquote>
      <figcaption>
          Ben Franklin, <cite>Poor Richard's Proverbs</cite>
      </figcaption>
  </figure>

  <p>Ben Franklin liked to say "<q>A penny saved is a penny earned</q>".</p>

The final major content tag to know is the <img> tag. This tag is used for inserting images into a page. It works like this:

<img src='gecko.jpg' alt="Gecko's rendering flow">

Where gecko.jpg is the path to the image, and Gecko’s rendering flow is the image’s “alt text”, which is the text that shows when you hover over the image.

There you go, you have now learned three basic but important content tags. Before we move on, let’s review:

  1. <ol> and <ul> are used to make lists. <li> is used for individual list items, which can contain anything.
  2. <blockquote> and <q> are used to markup block and inline quotations, respectively. Blockquotes should follow the example above.
  3. <img> is used to markup images. Make sure to provide both a path to the image and some sort of alernative text for accessibility.

Answer

You should see a numbered list, a dotted list, a nice big quote, a smaller quote, and whatever image you decided to use.

Form Tags

Another important building block of the web is the form. Forms are used day in and day out to collect information through a webpage that can then be processed, whether it’s for signing up for Facebook, posting to Twitter, or sharing on Pinterest.

The basic tag for forms in, unsurprisingly, the <form> tag. It indicates that the form fields contained within are associated together.

Every HTML form should have two attributes: action and method. action links to the page where the form data will be processed. method states what HTTP method will be used to transfer data between the pages. There are several different options, but the two most common are GET and POST. The first sends the data as text appended to a URL, the second as hidden data.

HTML5 introduced new, useful form inputs. Learn more about them from the HTML5 Doctor.

There are a number of different form fields, these include:

  1. <input>, a generic input field, it has several types:
    1. type='text', a text input field
    2. type='checkbox', a checkbox input field
    3. type='radio', a radio input; only one in a group may be selected at a time.
    4. type='password', a password input element. Autohides what is input into it.
    5. type='submit', similar to <button>.
    6. type='reset', resets the contents of the current form
    7. type='file', lets the user select a file to upload
    8. type='hidden', creates a hidden input field
  2. <button>, generally used for inputting the contents of a form
  3. <select>, creates a list of items to be selected
  4. <optgroup>, a group of options within a <select> element
  5. <option>, an individual option within a <select> element
  6. <textarea>, a large text input field

Those are the input types you should know.

Exercise 3.4

Try adding one of each of the above input types to a form on the page you’re working on. What do you see?

Answer

You should see a collection of different but recognizable form inputs. Make sure you know which one is associated with each HTML element.

Form accessibility is an important and complicated issue. Learn more from this study of form usability.

An important part of any form is making sure each field is labelled so that people know what the field is for. You do this with the <label> element, like so:

<label>
    Username: <input type='text' id='username' name='username'>
</label>

<label for='username'>Username</label>
<input type='text' id='username' name='username'>

The two examples given above are equivalent, although the first one is preferable, because it makes the label clickable the same way the form field is.

Beyond labelling your form fields, it is also important to set a tab order. This is done by giving each input element a tabindex attribute, like this:

<form action='processing.php' method='POST'>
    <label>Username: <input tabindex='1' type='text' name='username'></label>
    <label>Password: <input tabindex='2' type='text' name='password'></label>
    <label>Submit: <input tabindex='3' type='submit' name='submit'></label>
</form>

There are two more ways to give a form structure: the <fieldset> and <legend> elements. <fieldset> groups associated inputs together, and <legend> labels the fieldset, indicating what the grouping is.

There are two major attributes for form inputs, disabled and readonly. Disabled elements can’t be interacted with by the user. Readonly elements can be tabbed to, but can’t be modified.

The issue of processing a form is beyond the scope of this chapter. Just remember that the form itself is not the whole equation. Once the data has been filled out, it has to be processed to a program that can do something with it. HTML can’t do that, but languages like PHP, Javascript, Ruby, and Python can.

The “Div” Element

The final element you need to learn right now is also one of the most important: the <div> element. “div” stands for “division”, and that’s exactly what a <div> tag indicates: a semantic-less division in the document.

That may seem weird; haven’t you been learning that semantic code is important? Yes, it is. That’s why proper understanding of this tag is important. Unlike other tags, the <div> element relies on a good ID and/or class attribute to indicate its use in a document. You’ll learn more about those attributes in a bit, but for now just remember to never write a <div> without an ID and/or class!

Before moving on, let’s review:

  1. There are a number of different content tags
  2. Each content tag is for a specific type of content
  3. HTML forms are a complex but important part of the modern web
  4. The <div> element is useful but should be used smartly

Attributes

To learn more, check out the MDN list of HTML attributes.

You may have noticed that there was more than just the tags in the code examples above. Those other things are called “attributes”, and they are another important part of HTML. Attributes are modifiers of HTML elements. They can be used for styling an element, adding semantics to the element, improving the element's accessibility for people with disabilities, and lots of other things.

In the <img> tag example above, the tag has two attributes: src and alt. The first attribute tells the browser where the image is stored so that the browser can properly display it. The second attribute provides alternate text for the browser to display if it is unable to load the image.

The most important attributes to understand are the id and class attributes. Here’s how they work:

The “ID” Attribute

The id attribute is used to uniquely identify a single element in an HTML document. It can be used by both Javascript and CSS to access a specific element, and can also be appended to the end of a URL (preceded by an octothorpe) to directly access the element in your browser. It is incredibly useful. However, an ID can’t be reused. If something on a page has an ID of “header”, nothing else on the page can.

This is how an ID is set:

<div id='monkey'>
    <p>I like monkeys.</p>
</div>

The “Class” Attribute

The class attribute is used to identify a number of elements in an HTML document. Unlike IDs, Classes can be repeated as many times as desired per document, and exist largely as style hooks so that CSS can properly style elements on the page. Javascript can also attach events to Classes, but unlike with IDs, those events affect multiple elements on the page, instead of a single one.

This is how a class is set:

<div class='monkey'>
    <p>I still like monkeys.</p>
</div>

Notes on IDs and Classes

IDs and Classes may seem similar, but they are very different, and understanding how to use them properly is often the difference between a bad and good front end web developer. To make sure you do it right, follow the following rules of thumb:

  1. If you need a styling hook, use a class.
  2. If you need to access many elements in Javascript, use a class.
  3. If you need to access a single element in Javascript, use an ID.
  4. If you want an element to be accessible via a URL fragment identifier, use an ID.

Follow those rules and you should be good.

The “Alt” and “Title” Attributes

There are two other common and important attributes to understand: the alt and title attributes. Each of them is important for a page’s accessibility, and while they are very similar it is important to understand the differences in how they work.

Learn more about the alt attribute from this page about image accessibility.

The alt attribute provides alternative text if the element can’t be displayed. It is mostly used on the <img> tag, as images may not always be loaded (while text almost always is).

The title attribute provides supplementary information to the element it is applied to. This is useful in a number of situations (like the abbreviation definitions on this site). Titles are most often used on the <a> tag, to provide helpful information about whatever is being linked to.

Before we move on, let’s review:

  1. Classes and IDs are used as styling hooks
  2. IDs can be used as fragment identifiers in a URL
  3. alt and title attributes greatly enhance a page’s accessibility

Accessibility

One thing that many new HTML developers fail to consider is how accessible their site is. The web is used day in and day out by people with disabilities, and yet developers without impairments may often not consider whether their sites are usable for the blind, the deaf, the colorblind, or the disabled (to name a few).

To find out more about web accessibility, including how to implement it on your sites, go to the W3C’s Web Accessibility Initiative website.

The W3C has four main requirements for sites to be considered accessible. They are:

  1. Perceivable information and user interface
  2. Operable user interface and navigation
  3. Understandable information and user interface
  4. Robust content and reliable interpretation

Each of these has multiple sub-points, which we will address one at a time.

Perceivable Information and User Interface

The first requirement of an accessible site is that all content be perceivable to the user. For example, a blind user can’t view images, and so relies on a text description that their assistive software may read. This text description must adequately report the information given by the picture, so that the blind user is not receiving a limited or subpar experience because of their disability.

Whenever you are working on the web, particularly with non-text media, make sure to provide supplementary text information so that a person who is unable to view the picture/hear the audio/watch the video can still get the full amount of information those media provide.

Operable User Interface and Navigation

The next requirement of an accessible site is that a person be able to properly navigate the site using any software they may want to use. There are some users whose disabilities render them unable to use a mouse to navigate sites. These users rely on keyboard navigation to surf the web. When you are developing a site, if not enough care is given to making sure the site is keyboard-navigable, you may leave these users unable to use your site at all.

Beyond just making it possible to navigate your site using a keyboard, you want to give a lot of thought to the organization and design of that navigation. When developing sites for the average user you will often consider how usable something is, and those considerations will affect the placement of elements across your site. Give that same level of consideration to keyboard users, to make sure that they are able to easily and comfortably navigate your site as well.

Understandable Information and User Interface

The third requirement of an accessible site is that the information presented be understandable and easy to work with. This means that copy should be written in a way that minimizes confusion, has a logical flow to it, and provides definitions and other supporting information for jargon and other unclear language. It also means that document organization across pages should be consistent, predictable, and logical, and that ample opportunity be given for users submitting information to the site to review and correct their submissions.

One particular sticking point with web accessibility is the accessibility of forms. Forms are an important part of most people’s day-to-day use of the web, but if they are improperly labelled, keyboard tested, and reactive to user input they can be a nightmare for users utilizing assistive technology. For example, it has become increasingly common for web developers to place placeholder text inside a form field to indicate what information the field is looking for. This is all well and good for the average user, but placeholder text is not accessible for most assistive technology, and so a user using that technology may be left in the dark about what the field is even asking for. To combat this, make sure that all forms are properly labelled, and are organized in a clear and logical manner.

Robust Content and Reliable Interpretation

The fourth and final requirement of an accessible site is that content be comptable with current and future tools. This means that all documents created for the web should be consistent with current standards, and should utilize stable and well-defined specifications to make sure that support continues for the document’s features. When developing a site, always be wary of using features that are still in beta, and about following incomplete or changing specifications. Stability is important for a good website and for accessibility, and so always make sure to use the best, most stable technology available.

Before moving on, let’s review:

  1. Web accessibility is very important
  2. Accessibility must consider a broad range of impairments
  3. Accessibility can be complicated, and often requires careful consideration before implementation

Microformats

You now know that HTML is used to define the structure and content of a website. You’ve learned how HTML elements define the semantics of their contents, and you’ve learned how attributes can be used to enhance the usability of a site. However, websites are not only used by humans. Every day thousands of computers crawl the internet for sites like Google or Yandex, collecting information about the pages they crawl. This ability of computers to properly understand the contents of the webpage is of major importance to the web, and so when you creating a website you should not only be concerned with the usability of your site for humans, but for computers as well.

One issue computers encounter when parsing the contents of an HTML document is a lack of context or meaning attached to certain text. For example, if a Google bot is crawling a website to index it for search results, that bot may be interested in there is a phone number or street address on the website. If it can identify that content, it can be given special treatment in search results. However, that content may not always be obvious to the computer, and so without some help from the HTML document itself, it may be unable to properly identify that useful content.

Read more about Microformats on the Microformats website.

Enter Microformats. Microformats are part of an effort to create what is called the “Semantic Web”. The idea behind the Semantic Web is to take all of the largely unstructured webpages in existence today and convert into a web of machine-readable data. Microformats assist in this by providing small specifications that exist alongside HTML and are used to provide a greater level of semantic detail in describing content. There are a number of current Microformat specifications. Let’s take a look at one of the oldest ones: the hCard specification.

hCard exists to allow for the semantic markup of information pertaining to people and organizations. For example, many personal websites have an “About” page that contains contact information for the site owner. Without hCard, it may look something like this:

<p>My name is Dr. Gary Wiles and my phone number is (800) 123-4567</p>

This markup is reasonable from the perspective of the user. It provides the information it should provide and allows the user to contact the site owner. However, it lacks proper semantics for the computer. With hCard, the above code could be replaced with something like this:

<p class='vcard'>
    My name is
    <span class='fn'>
        <span class='honorific-prefix'>Dr.</span>
        <span class='given-name'>Gary</span>
        <span class='family-name'>Wiles</span>
    </span>
    and my phone number is <span class='tel'>(800) 123-4567</span>
</p>

Now, if a computer begins crawling the website it would have access to help semantic information for both the man’s name and his telephone number, allowing it to be properly processed.

Version 2.0 is the latest stable Microformats specification. Read it on the Microformats wiki.

Microformats are still a developing concept, and many new microformats are still in early stages. But the idea that computers should be given semantic information about content is an important one, and marking up your content to be properly accessible is always a smart move.

However, you may not always want content to be accessible. In fact sometimes, you may not want computers to understand it at all. In which case Microformats are definitely not the answer for you.

Before we move on, let’s review:

  1. Webpages should be accessible to computers as well as humans
  2. Microformats help computers process useful data
  3. Microformats make useful styling hooks to improve your webpages
  4. Microformats are still developing, but can be used right now

Scalable HTML

It can be easy to only look at the here and now when developing a website. You may be up against a deadline, whether it’s self-imposed, imposed by your boss, or a byproduct of the market you’re working in. During these times it can be easy to make quick decisions to finish a feature, decisions that are poorly considered and lead to problems down the road. While it may be tempting to ignore web standards and ignore good practices during these times, it is important to always consider the possibility for potential growth in a website.

These considerations manifest themselves in two key ways:

  1. Avoid overly generic ID attributes
  2. Don’t mix concerns

Let’s address these one by one.

Avoid Overly Generic ID Attribures

Let’s say you start a website and you are the only person working on it. You know that there are only a few key sections that you want to give IDs to. So you give them generic IDs like header and footer. As time goes on your site grows and you hire a couple of developers to help you run it. They begin to add features to the site, and they need some IDs for some Javascript code to hook into. They decide to use header as one of the IDs. They assume it’s not used elsewhere in the document, so they attach a Javascript even to it. Suddenly, people clicking things on the site are getting weird unexpected behavior. Your developers don’t know why and now have to spend time figuring out the bug, all because you both used a really generic ID.

When you are using IDs for Javascript hooks, try to make sure they aren’t overly generic. For example, if you have a button that opens and closes a modal window, don’t use the ID open or close or toggle for it. Use something clearer like modal-toggle. It’t longer, but it’s also clearer and less likely to conflict with future changes to the site.

Don’t Mix Concerns

Back in the old days of web development, it was incredibly common to see inline styles attached to HTML elements using the style attribute. It was fast, it was easy, and for many people it was the preferred way of styling a website.

This approach can be fine for one developer working on a small site, but it’s a nightmare to maintain, and leads to an incredibly tight coupling of structure and style. Whenever you want to make a change to the content of your site, you’ve also got to manually change a number of inline styles. Styles that aren’t commented, aren’t well organized, and aren’t always easy to find.

Modern web developers take care to separate their concerns. To make sure that the HTML documents they write are only for content, the CSS is only for style, and the Javascript is only for dynamic changes to the site. This makes things easier to find, easier to maintain, and easier to change when you decide to redesign.

Before we move on, let’s review:

  1. HTML is designed to be scalable
  2. HTML is for content, not style or dynamic behavior

Validation

If you’re interested in code validation, the W3C has a validator you can use freely.

Validation is the process of feeding your HTML code into some software that tells you whether your code conforms to current specifications or not. It is a useful tool in the arsenal of any web developer, but know that validation is not the most important part of your development process. Validation is best used as a tool to evaluate whether you are writing things in a clear and consistent manner. However, browser often implements features that are not yet part of a specification, and therefore do not evaluate as valid code when passed through a validator. This leaves you with a problem: do you remove the features the validator doesn’t like, or do you leave them in? The answer is to remember that while the specification exists to guide you and the browser makers, it’s browser themselves that define what you can or can’t use. If it works in the browsers you intend to support, and you believe your site is improved by it, then it is often a good decision to ignore the recommendations of the validator and keep the feature in.

  1. Validation is a useful tool, and nothing more

HTML5

You’ve now learned the basic language skills and concepts needed to be a modern web developer. You learned the history of HTML, you learned what elements are and how to use them, you learned about attributes and how they provide additional information for both computers and humans, you learned about accessibility and microformats, scalability and code validation. Now it’s time to learn about some of the new features being developed in the world of HTML.

As you found out in the History of HTML section of this chapter, the current version of the HTML specification is known as HTML5, and is being worked on in a collaboration between the WHATWG and the W3C.

There are currently a number of separate and interesting technologies being developed at part of the HTML5 specification. Here we will focus on single key development that affects the way you write markup day-to-day. It’s HTML5’s semantically richer tags.

In the Tags section of this chapter you learned about the basic HTML tags (their meaning and usage). These tags cover a wide array of different and important semantic scenarios, and have allowed developers to build semantically rich sites for over a decade. However, they do not cover all the cases, and for a long time even some common cases had to be covered using semantic-less <div>s and associated classes to provide meaning (a construct sometimes jokingly called “divitis”). One of the goals of HTML5 was to expand the number of tags provided by the language to make it easier to make semantically rich sites while avoiding unecessary classes.

The most eminently useful of the new elements are <section>, <article>, <header>, <footer>, <aside>, and <nav>.

The “section” Element

The <section> element indicates a logically connected group of content. This may sound pretty generic, but imagine the following code:

<div id='who-we-are' class='section'>
    <h2>
        <a href='#who-we-are' title='Who We Are'>
            Who We Are
        </a>
    </h2>
    <p>We are a team of rugby-ers who specialize in the knitting of doilies.</p>
</div>

In the above example, <div> is clearly part of a larger context (perhaps the About page for this rugby team’s website). The <div> is used both as a linking hook, and to indicate that the content it contains is all about a single topic: who they are as a group. Here’s the same idea, accomplished with the <section> element:

<section id='who-we-are'>
    <h2>
        <a href='#who-we-are' title='Who We Are'>
            Who We Are
        </a>
    </h2>
    <p>We are a team of rugby-ers who specialize in the knitting of doilies.</p>
</section>

In this case, the fact that the contained code is all logically grouped is indicates on the element, not the attribute, level of the code. This is easier to grok, clearer for machines, and cleaner. That’s what the <section> tag is for.

The “article” Element

The <article> element indicates a collection of self-contained content that is suitable for syndication (via an RSS or ATOM feed). This is great for news sites, blogs, and anything else that publishes content. Pre-HTML5, you would often see something like the following for a blog post:

<div class='post'>
    <h2>
        <a href='/posts/cheese' title='Certain Cheese Considered Delicious'>
            Certain Cheese Considered Delicious
        </a>
    </h2>
    <p>Seriously, cheese is awesome.</p>
    <div class='meta'>
        <p>This was posted on Aug 4th, 2013 by Don Wilberforce.</p>
    </div>
</div>

That can now be modified to this:

<article class='post'>
    <h2>
        <a href='/posts/cheese' title='Certain Cheese Considered Delicious'>
            Certain Cheese Considered Delicious
        </a>
    </h2>
    <p>Seriously, cheese is awesome.</p>
    <div class='footer meta'>
        <p>This was posted on Aug 4th, 2013 by Don Wilberforce.</p>
    </div>
</article>

I know what you’re thinking: that’s longer than the original! Yes, it is. However, it is also more semantically rich. Instead of relying on a fragile class to indicate that we’re looking at a blog post, we can look right at the tag. Furthermore, HTML parsers have no understanding of what class='post' means, to them it’s just a keyword to be matched to CSS styles. Now parsers and page scrapers (like those used by Instapaper, Pocket, Readability, and other read-it-later apps) can much more easily figure out where the post content is.

The “header” Element

The <header> element indicates a collection of introductory or navigational information. The most obvious use is the header of a webpage, which often has the name of the site, and some navigation items, maybe a search bar. Let’s see how that would look in pre-HTML4 markup:

<div class='header'>
    <h1 class='header-logo'>
        <a href='/' title='Wumbus: Social Effluvia'>
            Wumbus
        </a>
    </h1>
    <ul class='header-nav'>
        <li class='header-nav-item'>
            <a href='/about' title='About Us'>
                About
            </a>
        </li>
        <li class='header-nav-item'>
            <a href='/blog' title='Our Blog'>
                Blog
            </a>
        </li>
        <li class='header-nav-item'>
            <a href='/login' title='Login'>
                Login
            </a>
        </li>
        <li class='header-nav-item'>
            <a href='/join' title='Join'>
                Join
            </a>
        </li>
    </ul>
</div>

Here is with the <header> tag:

<header class='page-header'>
    <h1 class='page-header-logo'>
        <a href='/' title='Wumbus: Social Effluvia'>
            Wumbus
        </a>
    </h1>
    <ul class='page-header-nav'>
        <li class='page-header-nav-item'>
            <a href='/about' title='About Us'>
                About
            </a>
        </li>
        <li class='page-header-nav-item'>
            <a href='/blog' title='Our Blog'>
                Blog
            </a>
        </li>
        <li class='page-header-nav-item'>
            <a href='/login' title='Login'>
                Login
            </a>
        </li>
        <li class='page-header-nav-item'>
            <a href='/join' title='Join'>
                Join
            </a>
        </li>
    </ul>
</header>

“What the hell? It got bigger! Isn’t this worse than the code from before?” you may say. Well, let’s look at another example:

<article class='post'>
    <header class='post-header'>
        <h2>
            <a href='/posts/faust' title="Faust the Mouse's Daring Break!">
                Faust the Mouse's Daring Break!
            </a>
        </h2>
    </header>
    <p>It was unsuccessful.</p>
    <div class='footer meta'>
        <p>This was posted on May 15th, 2013 by Marsha Hyde.</p>
    </div>
</article>

That’s a blog post, and it’s got a header! You see, the <header> tag can be used over and over each document, and can be used in things like articles, sections, or whatever you feel needs a header. All the while it retains its semantic meaning, while still providing you will a tag you can hook into with your styles (and modify via classes, as always). That’s what’s so great about <header>.

The “Footer” Element

The <footer> element is much the same as the <header> element. It can be used in any context, whether at the page level, or within a <section> or <article>. There’s not much to say beyond that, so let’s just look at this example:

<article class='post'>
    <header class='post-header'>
        <h2>
            <a href='/posts/faust' title="Faust the Mouse's Daring Break!">
                Faust the Mouse's Daring Break!
            </a>
        </h2>
    </header>

    <p>It was unsuccessful.</p>

    <footer class='meta'>
        <p>This was posted on May 15th, 2013 by Marsha Hyde.</p>
    </footer>
</article>

As you can see, this is the same code as the last example, but with the footer replaced the same way the header was. That’s all there is, really.

The “Aside” Element

The <aside> element is where things get weird. It has two different meanings, depending on where in a document it is used. When inside of an <article> element, it is used to indicate content tangentially related to the main content. Outside of an <article> element, it indicates secondary content, like a sidebar. Here is an example that illustrates both uses:

<div class='container'>

    <header class='container-header'>
        <h1 class='header-logo'><a href='/' title='My Blog'>
                My Blog
        </a></h1>
        <nav class='header-nav'>
            <a href='nav-link' href='/about' title='About'>
                About
            </a>
            <a href='nav-link' href='/portfolio' title='Portfolio'>
                Portfolio
            </a>
        </nav>
    </header>

    <section class='posts'>
        <article class='post'>
            <header class='post-header'>
                <h2><a href='/hello-world' title='Hello World!'>
                        Hello World!
                </a></h2>
            </header>
            <aside>
                <p>"Hello World!" is a common phrase.</p>
            </aside>
            <p>This is an example post.</p>
            <footer class='post-footer'>
                <p>Posted on May 14th, 2013.</p>
            </footer>
        </article>
    </section>

    <aside class='sidebar'>
        <p>This blog is actually pretty terrible.</p>
    </aside>

    <footer class='page-footer'>
        <p>Copyright Martin Shoredale 2013.</p>
    </footer>
</div>

The “Nav” Element

You’ve already seen it a couple of times. The <nav> element indicates navigation, and should contain a collection of links. This can be done with a list (ordered or unordered) or without one. There are arguments for and against the listed approach. Read Chris Coyier’s post on the matter to learn more. Here’s an example of each approach:

<nav>
    <a href='/about' title='About'>
        About
    </a>
    <a href='/portfolio' title='Portfolio'>
        Portfolio
    </a>
</nav>

<nav>
    <ol>
        <li>
            <a href='/about' title='About'>
                About
            </a>
        </li>
        <li>
            <a href='/portfolio' title='Portfolio'>
                Portfolio
            </a>
        </li>
    </ol>
</nav>

One More Thing

While newer browsers support these semantically richer tags just fine, some older browsers (particularly IE8 and earlier), do not. There are ways to make these new tags work in older browsers, and it’s up to you to decide whether to use them. Just consider what browsers you want to support, and if those browsers include older models, you may want to consider sticking with the old markup for a while.

Before we move on, let’s review:

  1. HTML provides a number of semantically richer tags
  2. These tags can be used to replace semantically-null div tags
  3. These tags require a bit of effort to work in older browsers

Conclusion

So that is HTML. It is a relatively simple language, but is both flexible and powerful enough to be the backbone of the modern web. Before we move on to the rest of the body, let’s review:

  1. HTML is the product of multiple organizations with many contributors
  2. HTML is made up of tags, also called elements
  3. HTML elements provide both structure and semantics
  4. HTML elements can have attributes that modify them
  5. Good HTML code is scalable and accessible
  6. Accessibility is important for both humans and computers
  7. Recent developments in HTML have given us semantically richer tags
