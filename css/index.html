What Styles Are

CSS, as the name suggests, is a language for styling the content of a website. While HTML is about structure, CSS is about style, and the first step to understand CSS is understanding how its styles are applied to an HTML document.

You may remember our discussion of how webpages are constructed from way back in Chapter 2 (if you don’t, feel free to review). The browser takes the HTML code and constructs a DOM tree, which is the browser’s internal representation of a page’s structure. It then applies styles from any stylsheets the page is using to that DOM tree, followed by flowing and painting the page.

Selectors identify what HTML elements styles should be applied to. Each style within the brackets are applied to the elements that match the selector. The styles themselves are split into properties and values.

Selectors

There are several types of selectors:

  1. Element selectors
  2. Child selectors
  3. Adjacent sibling selectors
  4. Attribute selectors
  5. Class selectors
  6. ID selectors
  7. Pseudo-class selectors
  8. Pseudo-element selectors

Each matches elements in a specific way. Let’s cover them one-by-one.

Element Selectors

These are the simplest selectors. Element selectors directly match elements by name. For example, to style the <header> tag, you would use the header element selector. Element selectors are written as just the name of the element, like so:

p {
   margin: 1em 0;
   font-size: 16px;
   line-height: 1.5em;
}

Child Selectors

Child selectors style elements that match the selector to the right of the child operator, and are contained by the element to the left of the child operator. For example, the following code styles paragraphs contained by the <section> tag.

section > p {
    font-size: 24px;
    line-height: 1.5em;
}

Adjacent Sibling Selectors

Adjacent sibling selectors style elements that are next to, but not contained by, the element listed to the left of the operator. For example, the following code styles a paragraph following a header:

header + p {
    margin-top: -.5em;
}

Attribute Selectors

Attribute selectors style elements that have the given attribute and (possibly) value. For example, the following styles links that open in a new tab:

a[target="_blank"] {
    color: #c72f28;
}

There are a number of types of attribute selectors, they are:

Selector Behavior
[attribute="value"] Matches exact value of attribute
[attribute~="value"] Matches single value of a list of values
[attribute|="value"] Matches either exact value, or value followed by a hyphen
[attribute^="value"] Matches value starting with the given characters
[attribute$="value"] Matches value ending with the given characters
[attribute*="value"] Matches value containing the given characters

Class Selectors

Class selectors style elements based on their assigned class. The syntax of the selector is simply the class name, preceded by a period. For example, the following code styles any element with the .header class:

.header {
    background: #0381ff;
    color: #fff;
    padding: 1em;
}

Remember when we learned about class attributes last chapter? This is their main use: providing a styling hook for CSS.

ID Selectors

ID selectors are very similar to class selectors, but they style based on IDs. the following code styles an element with the ID #header:

#header {
    border-bottom: 1px solid #ccc;
    padding-bottom: 1em;
    margin-bottom: 1em;
}

You should generally avoid using ID selectors. They are less flexible than classes, and they can really mess with specificity (which we’ll cover in a bit).

Pseudo-Class Selectors

Pseudo-classes are a collection of oddball selector modifiers that can be used to style certain specific elements. Here are the important ones, along with what they modify:

Pseudo-Class Behavior
:link Styles a link that hasn’t been visited yet
:visited Styles a link that has been visited
:hover Sets styles for when the element is moused-over
:active Sets styles for when an element is being clicked
:focus Sets styles for when an element has been tabbed to or clicked
:target Sets styles for when an element is accessed via a fragment identifier
:enabled Sets styles for when an element is enabled
:disabled Sets styles for when an element is disabled
:checked Sets styles for when an element has been checked
:indeterminate Sets styles for when an element is neither unchecked or checked
:root Sets styles for the document’s root element. The <html> tag in HTML.
:first-child Sets styles for the first child of a given type
:last-child Sets styles for the last child of a given type
:nth-child(N) Sets styles for the children at indices matching the given formula
:nth-of-type(N) Sets styles for children of the given type at indices matching the given formula
:first-of-type Sets styles for the first child of a given type
:last-of-type Sets styles for the last child of a given type
:nth-last-of-type(N) Like :nth-of-type(N), but counting from the last to the first
:nth-last-child(N) Like :nth-child(N), but counting from the last to the first
:only-of-type Sets styles for element if it is the only child of its type
:not Reverses the condition of other pseudo-classes
:empty Sets styles for elements without contents
:lang Sets styles for elements with a given set language

Pseudo-class selectors can be combined, and together give you the ability to style a large number of interesting markup cases.

Pseudo-Element Selectors

Pseudo-element selectors are very similar to pseudo-class selectors, except they modify the DOM by inserting new nodes that can then be styled. The available pseudo-element selectors are:

Pseudo-element Behavior
:first-letter Styles the first letter of an element
:first-line Styles the first line of text in an element
::selection Styles the content when it is highlighted by the user
::before Creates a new element before the one being styled
::after Creates a new element after the one being styled

One thing to note is that both ::before and ::after must have a content property in order to appear in the document.

Properties and Values

You’ve now learned about selectors, but selectors are only part of the equation. The other parts are properties and values. Simply put: properties are the potential styles that be modified on an HTML element, and values indicate the currently used style. There are many many properties in CSS, and many more values, so it would be unreasonable to cover them all here. Instead we will cover those needed to make a basic website, and leave discovery of the others up to your exploration.

Positioning

The first important property to know is the position property. There are several possible position types in CSS. They are: static, relative, absolute, and fixed.

Static positioning is the default positioning of everything in an HTML element. Static elements are placed on the page according to their width, height, border, margin, padding, as well as those values for any adjacent or containing elements.

Relative positioning is the same as static positioning, except that relatively positioned elements may be adjusted manually using the top, right, bottom, and left properties.

Absolute positioning is the first big variation. Absolutely positioned elements are taken out of normal document flow, and are positioned using explicit offsets relative to the closest enclosing relatively positioned element.

Fixed positioning is like absolute positioning, but it is always positioned offset from the <body> element.

Display

Display is the next important property to understand. There are once again four main types: block, inline, inline-block, and none (there are others but we won’t cover them here).

Block display means that the element fills up the entire width of its containing element, and doesn’t have elements placed to the left or right of it on the page. Block positioned elements are affected by padding, margin, border, width, and height.

Inline display means that the element flows with adjacent elements and can be placed on the same line as them. Inline elements are not affected by margin, top/bottom padding, or width.

Inline-block display is similar to inline display, but the element is affected by margin, top/bottom padding, and width just like a block element is.

Non-displayed elements are taken out of the document entirely.

Margin, Padding, Border, Width, and Height

These five properties affect the size of an element. They each work slightly differently. The width and height of an element are exactly what they sound like. The padding of an element adds to the width and height according to the value it is set to. Padding directly affects width and height and the area added by padding receives the same background styles that the box does. Borders go around the padding, and receive whatever styles have been set for them. Margins go around the borders and are always transparent.

Font

There are a number of font styles, including font-weight, font-style, font-family, font-size, and line-height. Font weight determines the thickness of the text, font style sets whether the text is italicized, font family sets what typeface is used for the text, font size determines how large the text is, and line height sets the spacing above and below the text.

Color

Colors in CSS can be set a number of different ways. Every element can have a background color, a text color, a border color, a box shadow color, and a text shadow color. These colors can be set using hexadecimal notation, rgb, rgba, hsl, and keywords. For example, here is the color red set in all those ways:

#ff0000
rgb(255,0,0)
rgba(255,0,0,1)
hsl(0, 100, 50)
red

In CSS, all those color declarations are equivalent.

Notes on Style Matching

One important thing to understand about styles is the way they are matched to elements. While as English-speakers we generally read left-to-right, styles in CSS are matched right-to-left. For example in the following code, the browser first checks if an element is a paragraph. If it is, it checks if the paragraph’s parent is an <article>.

article p {
    color: #444;
}

To learn more read this article on selector performance by Harry Roberts.

This process means that some selectors have better performance than others. For example :last-child is significantly slower than :first-child. This is because :first-child only requires the browser to check previously indexed elements, while :last-child requires the browser to check an un-indexed element and then return to the original one.

The Cascade

The single most important concept to understand in CSS is the cascade. In CSS, any number of styles from any number of locations may apply to a single element, the cascade is a set of rules for deciding how conflicting styles are handled. Understanding it is paramount for the development of large sites where conflicts between styles are likely.

Before we get into the rules for the cascade, we must learn about specificity.

Specificity

Specificity is a numeric value given to each and every selector in CSS. Specificity is one of the criteria for resolving conflicts between styles. There are three levels of specificity, which we will assign the values 1, 10, and 100.

  1. Element selectors have a specificity of 1
  2. Class selectors have a specificity of 10
  3. ID selectors have a specificity of 100

When comparing conflicting styles, the one with the higher selector specificity wins. For example, in the following code the background of any <div> with a header class will be blue, not red.

div.header {
    background: blue;
}

.header {
    background: red;
}

One rule to follow is that you should generally try to minimize the specificity of your selectors. If you over-specify a selector early in a stylesheet you can develop problems when you try to overwrite those styles later on.

Rules for the Cascade

Now that you know what specificity is, let’s look at the specific rules for the cascade. Styles are set in the following order:

  1. Styles closer to the end of a file
  2. Styles in a file imported later
  3. Styles with higher specificity
  4. Styles with !important

Let’s look at some examples to clarify what these mean:

.nav {
    background: red;
}

.nav {
    background: blue;
}

In the example above, the background of any element with the class nav will be blue, because the blue background style came after the red background style.

In this example, the background of any element with the class nav will be green, because that style is from a stylesheet imported later than the other one.

.nav {
    background: white;
}

div.nav {
    background: orange;
}

In this example, the background will be orange, because the orange background style is set with a more specific selector.

.nav {
    background: white !important;
}

div.nav {
    background: orange;
}

In this example, the background will be white, because although the orange background style is set with a more specific selector, the specificity is overriden by the declaration of the less specific style as !important.

You should now have a general sense for how the cascade works. You should experiment with styles on your own to get a better feeling for it.

  1. CSS is based on the cascade: a flexible mechanism for applying conflicting styles
  2. Specificity is a major decider in style application
  3. Every selector has an associated specificity value
  4. There are a series of rules for the cascade, each of increasing weight
  5. !important is useful but dangerous. Use it only as a last resort.

Responsive Design

One of the increasingly important concepts in CSS is called “responsive design”. At its core, responsive design is about creating your styles in a way that can be easily accessed from any device: desktop computer, tablet, smartphone, etc. It is not a name for a specific technology, but rather a group of technologies and concepts that make the development of cross-browser websites possible. We will not cover the entirety of responsive design here, but will explore the basic ideas and methods needs to make a site responsive.

Start With a Fluid Grid

For an in-depth look at fluid grids read Ethan Marcotte’s seminal work on the subject.

Responsive web design can be considered a series of layered concepts and technologies that build to make a truly universal site. The first layer is making your site fluid. The general concepts of a fluid site are:

  1. Start with a static grid
  2. Measure type in relative units
  3. Redo the grid you want in relative units

What do we mean by relative units? Well, you’ve already seen things measured in pixels. Pixels are the basic unit of measure on the web. They represent the smallest possible division of the screen. They are relatively constant across devices. However, they are also completely inflexible. Relative units are units that are measured relative to something else. The major relative units are ems, rems, and percentages. Before we move on to the grid itself, let’s explain each of them.

Relative Units

ems are a relative unit measured in the following way. A single em is equal to the desired pixel font size divided by the font size of the closest parent. In the following example, font-size: 1.5em is equal to 24px because the font size of the parent element in 16px.

section {
    font-size: 16px;
    line-height: 1.5em;
}

    section p {
        font-size: 1.5em;
        line-height: 1.5em;
    }

But wait, what’s up with the other 1.5em values? What are they equal to? Well, here’s the same code as before, commented with the pixel values of each relative unit.

section {
    font-size: 16px;
    line-height: 1.5em; /* 24px */
}

    section p {
        font-size: 1.5em; /* 24px */
        line-height: 1.5em; /* 36px */
    }

For line-height, ems are set relative to the font-size of the current element, not the closest parent. It’s a little detail, but it’t an important one to remember.

rems are almost identical to ems, except that instead of being measured relative to the closest parent, they are set relative to the <html> element’s font size. They avoid a common issue with ems, which is that if you accidentally put an element that is sized using ems in another one also sized with them, the font size of the child element will be bigger than expected (because the ems compound each other). rems are newer, and have reasonably good cross-browser support, but it’s up to you to decide whether to use them in your projects.

The final relative measurement is percentages, which work just like ems, except the values are multiplied by 100 (just like when you’re converting from a number with a decimal point to a percentage in real life). Here’s the same code as before, but with percentages:

section {
    font-size: 16px;
    line-height: 150%;
}

    section p {
        font-size: 150%;
        line-height: 150%;
    }

Percentages are generally more tedious to write for font sizes, and are usually reserved for things like width, height, margin, and padding.

The Actual Grid

Now that you know the various units of relative measure, let’s discuss the creation of a fluid grid. The idea behind a fluid grid, is to take a static grid, and turn all the static measurements into relative ones, thus allowing the grid to shrink and grow in an organic and friendly way. That’s it. Just make a site with static measurements, swap out the pixels for ems, rems, and percentages, and you’re done.

Talking to the User

The fluid grid approach is not perfect. It works for screen sizes relatively close to the size of the static site, but for much smaller or much larger screens you start to develop issues. What you need is a way to adapt the CSS to things like the size of the window. You need media queries.

Media queries are a relatively new CSS technology you can use to conditionally set styles based on things like pixel density and window width. Here’s an example stylesheet, with a media query that adapts the site to a smaller screen.

body {
    font-size: 16px;
    line-height: 1.5em;
    background: #fff;
    color: #444;
    font-family: Helvetica, Arial, sans-serif;
}

    section {
        width: 500px;
        margin: 4em auto;
    }

@media screen and (max-width: 500px) {
    section {
        width: 90%;
        margin: 2em auto;
    }
}

Being mobile first is a really good idea. Learn more from Luke Wroblewski, the guy who wrote the book on the subject.

That’s the idea. Now you may notice that in the example above the base code is for a desktop site. That’s all well and good, but as you develop increasingly complicated sites you can encounter problems with that approach. The styles needed for a mobile site are generally simpler than those needed for a desktop one. If you start with the desktop site, you end up having to override a lot of styles for the smaller screens. If you start with the mobile design you can simply build on it to reach the desktop one.

Still More To It

Those are the basic ideas of responsive design. There’s a lot more to learn, but the information presented here should be enough to get you started.

  1. Responsive design a relatively new concept that tries to provide a strong web experience to users of any and every device.
  2. The precursor to responsive design was the fluid grid, which uses relative measurements to introduce flexibility to a site’s design
  3. Media queries allow you to go further and actually apply styles based on the nature of the device accessing the site
  4. Because it’s relatively new, responsive design is still developing, and there’s still a lot more to learn

Preprocessors

You may have noticed during this chapter that CSS is a relatively limited language. It provides all the bare essentials needed for styling a website, but is very static and has no good system for handling abstractions. Other CSS developers noticed this too, and for the last several years a large amount of effort has been expended toward developing systems to improve or build upon CSS. The end result is a group of systems called “CSS Preprocessors”.

At their core CSS preprocessors are distinct languages that are compiled into CSS. They provide useful abstractions and tools to speed up the development of CSS, and because they become CSS in the end they require no weird shoehorning workarounds to use today.

The two most popular preprocessors right now are SASS and LESS. They are very similar, but with some key differences. The most important difference is that SASS runs in a Ruby-based environment (meaning it is written in Ruby, requires Ruby to run, and integrates with Ruby systems) and LESS runs in a Javascript-based environment (meaning it is written in Javascript, requires Javascript to run, and integrates with Javascript systems).

We will cover both SASS and LESS in this section. They are both strong tools, so which one you choose is entirely up to you.

SASS

See the full SASS spec at sass-lang.com.

As previously mentioned, SASS is Ruby-based. It offers a number of features to help you build quality sites quickly. Here is an example of the first one: nesting.

section {
    padding: 1em;
    color: #444;

    p {
        font-size: 16px;
        line-height: 1.5em;
    }
}

That SASS code becomes the following CSS code:

section {
    padding: 1em;
    color: #444;
}

section p {
    font-size: 16px;
    line-height: 1.5em;
}

As you can see, the SASS code is simpler and cleaner. Here’s another example showing off another cool nesting feature:

a {
    color: #c72f28;

    &:hover,
    &:focus {
        color: #0381ff;
    }
}

Which becomes the following:

a {
    color: #c72f28;
}

a:hover,
a:focus {
    color: #0381ff;
}

In this code, the ampersand is used as a placeholder for the containing selector, allowing you to do things like you just read.

Another simple but useful feature is variables. Instead of manually settings things like color and font size across a number of elements, you can define a variable and just reference the variable when you want to use it. Like so:

$text-color: #444;

body {
    color: $text-color;
}

You can also do mixins, which allow you to do stuff like this:

@mixin box-sizing($sizing: border-box) {
    -webkit-box-sizing: $sizing;
       -moz-box-sizing: $sizing;
            box-sizing: $sizing;
}

* {
    @include box-sizing(border-box);
}

Which becomes:

* {
    -webkit-box-sizing: border-box;
       -moz-box-sizing: border-box;
            box-sizing: border-box;
}

But could just as easily become:

* {
    -webkit-box-sizing: content-box;
       -moz-box-sizing: content-box;
            box-sizing: content-box;
}

Just by passing in a different value to the mixin!

As you can see, there are a number of different things available in SASS. It’s not important that you learn all of them right now. Just having awareness of this tool can be a big help, and you may opt to try it on a future project of yours. Of course, you may opt for LESS instead.

LESS

See the full LESS spec at lesscss.org.

As previously mentioned, LESS is Javascript-based. It is incredibly similar to SASS, with just a few key differences that’ll be covered here.

The first major difference is that variables are prefaced with an “@” symbol instead of the “$” used by SASS. Here’s an example you saw for SASS earlier, written in LESS.

@text-color: #444;

body {
    color: @text-color;
}

That turns into this:

body {
    color: #444;
}

Just like the SASS code did.

Another big difference is that unlike SASS, LESS mixins do not have to be prefaced with @mixin. Here is another repeat example, once again in LESS instead of SASS:

.box-sizing(@sizing: border-box) {
    -webkit-box-sizing: @sizing;
       -moz-box-sizing: @sizing;
            box-sizing: @sizing;
}

* {
    .box-sizing(border-box);
}

Those are the main differences you should be aware of right now. Like SASS, LESS has a lot more going on that has been shown here. If you’re interested in learning more just check out the websites for SASS and LESS. They both have incredibly thorough documentation to guide you through learning everything these CSS preprocessors can do.

  1. CSS Preprocessors are languages that are turned into CSS
  2. Preprocessors provide useful features that aren’t available in CSS to help you write code faster and better
  3. SASS and LESS are the two most popular preprocessors, and they are both very similar
  4. Nesting, variables, and mixins are just the basics of what SASS and LESS offer

Scalable CSS

As you develop larger and more complicated websites, your CSS will likely grow a large amount. One of the challenges of large scale web development is knowing how to keep your stylesheets organized and easy to maintain. There are several popular methodoligies that provide a set of concepts and rules for organizing and structuring your styles: OOCSS, BEM, and SMACSS. In this section we will cover each of them; describing how they work, what’s good about them, and what problems you may encounter when following the paradigm.

Same as with the preprocessors, the choice of what methodology to use is yours.

OOCSS

OOCSS is, understandably, all about Objects. It has two main concepts:

  1. Structure should be separated from skin
  2. Content should be separated from container

What the first one means is that you should separate the styles that determine placement on a page from those styles that deterine appearance. The second one means that instead of using the decendent selector (which is fragile and tied to the current content of a container), we should use generic class names that can be applied to whatever elements we want to style.

Let’s look at a couple of examples. Here is some CSS not designed with the OOCSS paradigm in mind:

.button {
  width: 20em;
  height: 10em;
  padding: 1em;
  color: #444;
  font-family: Helvetica, Arial, sans-serif;
}

.card {
  width: 15em;
  overflow: hidden;
  color: #444;
  font-family: Helvetica, Arial, sans-serif;
}

Here it is rewritten using OOCSS:

.button {
  width: 20em;
  height: 10em;
  padding: 1em;
}

.card {
  width: 15em;
  overflow: hidden;
}

.skin {
  color: #444;
  font-family: Helvetica, Arial, sans-serif;
}

As you can see, while there is now an additional class, all the classes are more generic and reusable. If we create a new class later in the project and want it to the have the same color and font as the others, we have a class that does just that ready to go. Here’s another example:

.header h1 {
    font-size: 2em;
    line-height: 1.5em;
    font-weight: 700;
    font-family: Georgia, "Times New Roman", Times, serif;
}

Here it is rewritten with OOCSS:

.header-title {
    font-size: 2em;
    line-height: 1.5em;
    font-weight: 700;
    font-family: Georgia, "Times New Roman", Times, serif;
}

As you can see, .header-title has the same styles, but it is more flexible because it is not tied to a specific element.

Advantages

So what are the advantages of OOCSS? Well, they are:

  1. It’s simple
  2. It encourages code reuse
  3. It avoids HTML/CSS coupling

Disadvantages

The disadvantages are:

  1. It’s overly vague
  2. It encourages Class-bloat in your HTML
  3. It does not prescribe CSS structure on a broader scale

BEM

In BEM, the components of a webpage are split into three types. They are:

  1. Blocks: The major sections of the page
  2. Elements: The parts that make up a block
  3. Modifiers: The variations of elements

Blocks are independent elements of a page that can be mixed and matched. They serve a single function, and their contents can be considered to be associated or logically connected.

Elements are things that only make sense within the context of a containing block. For example, you may have a “Search” block with an input element. Outside of the block the input element can mean anything, but in the context of the Search block it is clearly used for searching. That’s the idea of elements.

Modifiers are classes that can be applied to elements and blocks to vary them as needed. They are associated to a specific element or block and do not stand on their own.

The interesting syntactic distinction developed for BEM is the way that classes are written. In standard BEM (although you are free to follow another pattern), the following standards exist for class creation:

  1. Elements are separated from their containing block by two underscores
  2. Modifiers are separated from their associated element or block by two hyphens

So, for example, a class name for an input element within a search bar with a modifier to float it to the left might be search__input--left. This may seem overly verbose, but the advantage is that the class names clearly indicate exactly what is going on. They are incredibly descriptive, and if you can handle the verbosity they may be the right choice for your project.

Advantages

  1. Clear rules for class naming
  2. Provides rules for file structure (given in the documentation)
  3. Flexible

Disadvantages

  1. Overly verbose
  2. Too focused on Javascript
  3. Optimized for machines, not humans

SMACSS

The final paradigm we will cover is called SMACSS. SMACSS breaks up styles into five distinct groups: base, layout, module, state, and theme.

The “base” group is for styles that are consistent across the site, and are largely restricted to styles using element selectors and pseudo-class selectors.

The “layout” group is for styles that define the large scale layout of the site.

The “module” group is for styles defining individual components of the site. These are the portions of the site that are layed out on the page by the layout styles. Their state changes are not defined internally, but are instead defined in the next section, which is&hellip

“State” is the section for state changes, particular those that are tied to Javascript code. Putting all the Javascript-associated CSS code in one plane helps with future updating when you want to update Javascript functionality. Instead of fishing through styles all the dependent code is in one place.

“Theme”, the final group, is optional, but is used for styles that define a theme that sits on top of the site. This is common for things like blogs where the user has the ability to restyle their site. You may or may not have a need for it on your projects.

Advantages

  1. Flexible
  2. Not necessarily verbose
  3. Keeps styles clearly organized by usage

Disadvantages

  1. Layout and module aren’t always distinct
  2. Theme is an unecessary section that shouldn’t be a main section
  3. It is overly restrictive for folder organization
  1. As sites grow it becomes increasingly important to have a method to manage your CSS code
  2. In the last few years a number of such methods have developed. OOCSS, BEM, and SMACSS are the most popular of them
  3. OOCSS treats everything objects, and encourages good practices like separation of content from style
  4. BEM’s strict rules and syntax can be verbose, but are full of semantic meaning
  5. SMACSS is a mix of flexible and strict. It encourages a specific folder structure in the interest of being predictable

Conclusion

As you can see, like HTML, CSS is a simple language with a surprising level of nuance. This section has not covered the entirety of the language, and you are seriously encouraged to go and experiment and research things on your own.

  1. CSS is based on selectors, properties, and values
  2. All selectors have a specificity value, which affects how they cascade
  3. Responsive design methodologies allow you to develop universal sites by directly querying the device accessing the page
  4. As sites grow, scalable CSS becomes an increasingly important issue. There are a number of different methodologies, each with their own advantages and disadvantages
